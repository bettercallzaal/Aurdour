# Contributing Guide

Thanks for your interest in contributing to this open-source audio player project!

## 🎯 Philosophy

This project is about creating **ownable, composable, social-native audio experiences**. Keep this in mind when proposing features or changes.

## 🚀 Getting Started

1. Fork the repository
2. Clone your fork: `git clone https://github.com/yourusername/ardour-web-player.git`
3. Create a branch: `git checkout -b feature/your-feature-name`
4. Make your changes
5. Test locally: `cd web && python3 -m http.server 8000`
6. Commit: `git commit -m "Add: your feature description"`
7. Push: `git push origin feature/your-feature-name`
8. Open a Pull Request

### Testing
- Browser compatibility testing
- Mobile device testing
- Accessibility testing

## 🎨 Code Style

### JavaScript
- Use ES6+ features
- Prefer `const` over `let`
- Use descriptive variable names
- Add comments for complex logic
- Follow existing code structure

### CSS
- Use CSS variables for theming
- Mobile-first responsive design
- Follow BEM naming convention
- Keep specificity low

### Python
- Follow PEP 8
- Use type hints
- Add docstrings
- Keep functions focused

## 🧪 Testing

Before submitting:
- [ ] Test in Chrome, Firefox, Safari
- [ ] Test on mobile devices
- [ ] Verify keyboard navigation
- [ ] Check console for errors
- [ ] Test with various audio formats
- [ ] Validate JSON schema

## 📝 Commit Messages

Format: `Type: Brief description`

**Types:**
- `Add:` New feature
- `Fix:` Bug fix
- `Update:` Modify existing feature
- `Remove:` Delete code/feature
- `Docs:` Documentation only
- `Style:` Formatting, no code change
- `Refactor:` Code restructuring
- `Test:` Add/update tests
- `Perf:` Performance improvement

**Examples:**
```
Add: Playback speed control
Fix: Marker highlighting on mobile
Update: Improve waveform rendering performance
Docs: Add deployment guide for Cloudflare Pages
```

## 🐛 Reporting Bugs

Include:
- Browser and version
- Operating system
- Steps to reproduce
- Expected behavior
- Actual behavior
- Screenshots/videos if applicable
- Console errors

## 💡 Proposing Features

Before opening an issue:
1. Check existing issues/PRs
2. Review EXTENSIONS.md
3. Consider if it aligns with project philosophy
4. Think about implementation complexity

In your proposal:
- Describe the feature
- Explain the use case
- Suggest implementation approach
- Consider backwards compatibility

## 🔍 Code Review Process

PRs will be reviewed for:

- Accessibility
- Documentation
- Test coverage
